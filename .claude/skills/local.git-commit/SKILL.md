# /git-commit スキル

## 手順

### ステップ1: 現状把握

以下のコマンドを**並列**で実行:

- `git status`（未追跡ファイル確認。`-uall`は使わない）
- `git diff --staged`（ステージ済みの変更内容）
- `git diff`（未ステージの変更内容）
- `git log --oneline -20`（直近のコミットメッセージのprefixスタイル・**言語**参照用）
- `git branch --show-current`（現在のブランチ名）

### ステップ2: ブランチ判定

- **main/masterブランチの場合**: ブランチ作成が必要
- **それ以外のブランチの場合**: そのまま続行

ブランチ名のルール:
- `/` は使わない。`-` で区切る（例: `fix-typo-in-readme`, `feat-add-login-page`）

### ステップ3: ステージング判定

ステージ済みファイルがない場合:
- 会話の文脈や実装状況から、ステージすべきファイルを判断する
- `.env`、認証情報等のファイルが含まれる場合は警告する

ステージ済みファイルがある場合はそのまま続行。

### ステップ4: コミットメッセージ生成

変更理由が不明な場合はユーザーに質問する。

#### 基本原則: 「なぜ」を記載する

変更内容（what）は `git diff` で確認できるが、変更理由（why）はコミットメッセージにしか記録できない。diffから読み取れる情報の優先度は低い。**会話の文脈**から以下を読み取りメッセージの主軸にする:

- **何が問題だったのか**（背景・課題）
- **このコミットでどう改善されるのか**（効果・目的）

typo修正などの自明な変更を除き、**必ず変更が必要だった理由を記載する。**

#### 主観的な判断には客観的根拠を添える

「不要」「不適切」「古い」「冗長」などの主観的な判断を含むコミットメッセージは、将来の開発者がその妥当性を検証できない。**なぜそう判断したのか**の根拠を必ず記載する。

❌ 悪い例:
```
不要なコードを削除
```

✅ 良い例:
```
未使用のUserSerializer classを削除

grepで検索した結果、プロジェクト内に呼び出し元が存在しない。
2024年のAPI v2移行時に旧エンドポイントと共に不要になったもの。
```

根拠として有効な情報の例:
- **検索結果**: 「grepで呼び出し元がない」「参照箇所が0件」
- **意思決定の経緯**: 「MTGで使用しないことが決まった」「issueで廃止が合意された」
- **技術的事実**: 「この機能はv2で別の実装に置き換えられた」「このライブラリはEOLを迎えた」
- **エラーや不具合の証拠**: 「本番でこのエラーが発生している」「CIでこのテストが失敗する」
- **数値データ**: 「レスポンスタイムが200ms→50msに改善」「メモリ使用量が16MB削減」

#### コミットメッセージの構造

```
1行目：[任意 型(スコープ): ]変更内容の簡潔な要約（50文字以内推奨）

3行目以降：動機・背景（なぜこの変更が必要だったか）
```

- **1行目**: 50文字以内を推奨（最大72文字）、命令形（「〜を追加」「〜を修正」「〜を削除」）、末尾に句読点は不要
- **2行目**: 空行
- **3行目以降**: 動機・背景を簡潔に説明（自明な変更を除き必須）。72文字程度で改行。関連するissue/PR番号があれば含める

#### 本文の取捨選択ルール

本文の各文を書く前に「この情報は diff を読めばわかるか？」を判断する:

- **diff から自明 → 書かない**: 具体的なコード変更の説明（何を追加した、どのオプションを指定した等）
- **diff からは読み取れない → 書く**: 変更が必要になった動機・背景・課題
- **diff が複雑で全体像が掴みにくい → 書いてよい**: 変更の概要（ただし簡潔に）

#### 本文が省略できる場合

変更内容が1行のサマリーで完全に理解でき、「なぜ」が自明な場合のみ省略可:
- typo修正
- lint/フォーマット修正

「不要コード削除」は本文省略不可。なぜ不要と判断したかの根拠が必要。

#### Conventional Commits prefix

##### 事前判定（必須）

メッセージを書き始める前に、ステップ1で取得した `git log --oneline -20` を必ず確認し、以下の手順で prefix を使うかを判定する:

1. 直近20件のうち、Conventional Commits prefix（`feat:`, `fix:`, `chore:`, `refactor:`, `perf:`, `style:`, `test:`, `docs:`, `ci:`, `build:` など）が使われているコミット数をカウントする
2. **過半数（11件以上）で prefix が使われている場合のみ** prefix を使う
3. それ未満の場合は **prefix を付けてはならない**（「任意」ではなく禁止）

❌ 悪い例（直近ログで prefix を使っていないリポジトリの場合）:

```
feat: ログイン画面を追加
fix: 認証エラーのハンドリングを修正
```

✅ 良い例（同じリポジトリ）:

```
ログイン画面を追加
認証エラーのハンドリングを修正
```

学習傾向として Conventional Commits を付けがちだが、リポジトリの既存スタイルに合わない prefix は履歴の一貫性を損なう。判定結果に従うこと。

##### prefix を使う場合の型選択

事前判定で「prefix を使う」と判定した場合のみ、以下のフローチャートで型を選ぶ:

```
ユーザーから見て新しい機能・振る舞いが追加される？
  → Yes → feat
  → No  → ユーザーが困っている不具合を修正した？
            → Yes → fix
            → No  → chore（より具体的な型があればそちらを使ってもよい）
```

判断基準:
- `feat`: リリースノートに「新機能」として書けるか？
- `fix`: ユーザーやシステムが困っている不具合があるか？
- `chore`: リリースノートに書く必要がない裏方の作業か？

その他の型（任意）: `refactor`, `perf`, `style`, `test`, `docs`, `ci`, `build`

破壊的変更がある場合は型の後に `!` を付ける（例: `feat!: APIレスポンスの形式を変更`）

#### 過去のコミットログの参照について

過去のコミットログは**prefixスタイルの参照のみ**に使う。過去のコミットメッセージの品質が高い保証はないため、メッセージの内容・詳細度・書き方を過去に合わせてはならない。常にこのスキルのルールに従って高品質なメッセージを生成する。

#### その他のルール

- **コミットメッセージの言語**: リポジトリの既存コミットログに合わせる
  - 判定方法: `git log --oneline -20` で直近のコミットメッセージを確認
    - 大半が日本語 → **日本語**で記述（技術用語は英語のまま使ってよい）
    - 大半が英語 → **英語**で記述
    - 混在・初回コミット等で判定不能の場合 → README やソースコード内のコメント言語から推測。それでも不明なら**英語**をデフォルトとする
  - ユーザーから明示的な指示があればそれを優先する
- **技術用語の表記**: 公式表記を遵守する（GitHub, RuboCop, TypeScript, Next.js, PostgreSQL, nginx 等）
- **変数名・関数名・コマンド名などのコード要素**: バッククォート（`` ` ``）で囲む（例: `USE_IAM_PROFILE`、`aws_access_key_id`）。GitHubのコミット詳細画面でMarkdownとしてレンダリングされ、可読性が向上する
- **コミット粒度**: 独立してリバートできるアトミックな単位。変更理由ごとにコミットを分ける。ステージ済みの変更が複数の論理的変更を含む場合、コミットの分割を提案する

### ステップ5: ユーザー確認（1回のみ）

以下の内容をまとめて1回だけ確認する:

- **ブランチ**: 新規作成する場合はブランチ名、既存ブランチの場合はそのブランチ名
- **ステージ対象**: ステージするファイル一覧（既にステージ済みの場合はその旨）
- **コミットメッセージ**: 生成したメッセージ全文

ユーザーの承認後、以下を順次実行:
1. ブランチ作成（必要な場合）: `git switch -c {ブランチ名}`
2. ファイルのステージ（必要な場合）: `git add` で個別にステージ
3. コミット実行（HEREDOC形式）:

```bash
git commit -m "$(cat <<'EOF'
コミットメッセージ1行目

本文（必要な場合）
EOF
)"
```

### ステップ6: hook失敗時の対応

pre-commit hookが失敗した場合:

1. エラー内容を分析
2. 問題を修正
3. 修正ファイルを再ステージ
4. **新規コミット**を作成（`--amend` は絶対に使わない。hookが失敗した場合コミットは作成されていないため、`--amend`は前のコミットを書き換えてしまう）

## 禁止事項

- `git push` はこのスキル内では実行しない（ユーザーが明示的に依頼した場合のみ）
- `--amend` は使わない
- `--no-verify` は使わない
- `.env`、認証情報、シークレットファイルのコミットは警告する
