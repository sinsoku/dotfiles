# レビュー観点（汎用・全プロジェクト共通）

`local.review` が読み込む**汎用観点の真実源**。全プロジェクトで満たすべき観点だけを置く。
公式 `pr-review-toolkit:*` Agent が見られる観点は**その Agent に委譲**し、ここに独自定義しない。

観点は次を重ねて使う（後ほど優先）:

1. **共通**（このファイル）— 全プロジェクト共通。
2. **プロジェクト** `<project_root>/.claude/REVIEW.md` — 技術スタック固有・ドメイン固有（チーム共有）。
3. **開発者** `<project_root>/.claude/REVIEW.local.md` — 個人の観点（gitignore・共有しない）。

---

## レンズ登録

ルータは `when` と差分を突き合わせて起動レンズを選ぶ。`handler` が `pr-review-toolkit:*` なら**公式 Agent をそのままディスパッチ**（独自レビューを書かない）、`custom` なら公式 Agent が無い観点で**本文を下の「カスタムレンズ観点」から注入**する。

| name | 観点 | when（起動条件） | handler |
|---|---|---|---|
| `correctness` | 正しさ・ロジック・品質・規約 | always | `pr-review-toolkit:code-reviewer` |
| `simplification` | 単純化・可読性 | 実装コード（ロジック）を追加・変更したとき | `pr-review-toolkit:code-simplifier` |
| `silent-failure` | 握り潰し・エラー処理 | rescue/catch/begin・nil/空フォールバックを追加・変更したとき | `pr-review-toolkit:silent-failure-hunter` |
| `type-design` | 型設計 | 型・class・struct・enum・interface を新規・変更したとき | `pr-review-toolkit:type-design-analyzer` |
| `test-coverage` | テスト網羅 | ロジックを追加・変更したとき（挙動不変の純リファクタは除く） | `pr-review-toolkit:pr-test-analyzer` |
| `comments` | コメント整合 | コメント・docstring を追加・変更したとき | `pr-review-toolkit:comment-analyzer` |
| `caller-impact` | 削除/リネーム/シグネチャ変更が呼び出し元を壊す | メソッド/関数/クラス/定数/公開 API の削除・リネーム・シグネチャ変更があるとき | `custom` |
| `deploy-impact` | DB 破壊的変更 × ローリングデプロイの順序安全性 | カラム/テーブル削除・NOT NULL 追加・型/enum 変更・データ移行を伴う migration があるとき | `custom` |
| `consumer-impact` | 値/意味の変化で consumer の判定がズレる | 既存フィールドの値・shape・意味が変化したとき（contract_change_c） | `custom` |
| `devils-advocate` | 設計前提の批判 | always | `custom` |
| `communication` | タイトル・説明・コミットメッセージの文章品質 | always | `custom` |

> 起動条件は機械的トリガーだが、最終判断は「そのレンズにレビュー対象（該当差分）が存在するか」。存在しなければ起動しない。迷ったら起動する（false negative の方が高コスト）。

## カスタムレンズ観点（`handler=custom` のレンズに注入する本文）

> `caller-impact` / `deploy-impact` / `consumer-impact` は**変更の波及を異なる軸で見る同族3レンズ**（存在・シグネチャ→呼び出し元 ／ スキーマ→デプロイ共存 ／ 値・意味→consumer 判定）。devils-advocate は波及ではなく変更そのものの妥当性を見る。

### caller-impact（呼び出し元への影響）
コードシンボル（メソッド/関数/クラス/定数/公開 API）の削除・リネーム・シグネチャ変更が、change set の外の呼び出し元を壊さないか。
- 削除/リネームした要素を参照する箇所が他に残っていないか（残れば未変更でも実行時に壊れる＝NoMethodError 等）。変更行外でも `CHANGE_CAUSED`。
- 追加は影響が小さいが、削除・リネームは「全呼び出し元を確認したか」を前提に厳しく見る。
- シグネチャ変更（引数・戻り値）は既存呼び出し元との互換を確認する。
※ DB カラム/テーブルの削除は deploy-impact、値・意味の変化は consumer-impact が担当（直交）。

### deploy-impact（デプロイ共存への影響）
DB migration の後に Web サーバをローリングアップデートするため、移行中は新旧スキーマと新旧アプリが一時的に共存する。破壊的変更がこの共存期間で壊れないかを expand/contract の観点で見る。
- テーブル/カラム削除は、**先行リリースでアプリ側の参照を全て除去してから**、別リリースで `drop` する手順になっているか。同一 PR で「参照削除＋drop」をまとめると、migration 適用後・旧アプリ稼働中に壊れる。
- NOT NULL 追加・型変更・enum 変更などは、事前のデータ移行（バックフィル）が全デプロイ環境で完了しているか。未完了の環境では migration が失敗する。
- 移行中、旧アプリ（変更前コード）×新スキーマ、新アプリ×旧スキーマがそれぞれ共存可能か。

### consumer-impact（consumer の判定への影響）
変更が**既存フィールド（API 属性 / シリアライザ出力 / 公開メソッドの戻り値）の値・shape・意味**を変えたとき、
そのフィールドを読む consumer を網羅し、判定基準のドリフトを検出する。
**consumer のコードが変わらず壊れていなくても**、新しい真実源と consumer 側の判定軸が乖離していれば指摘する
（変更が症状を偶然マスクするだけで consumer のロジックは依然誤り、というケースも含む）。
「単一の真実源に統一」と謳う変更は、同じ判定が別箇所で別ロジックとして重複していないかを横断確認する。

### devils-advocate（設計前提の批判）
変更の前提と過不足を疑う。コードの正しさではなく**変更全体の妥当性**を問う。
- **最小の差分**: 課題（Why）を満たす過不足ない最小の変更か。無関係な「ついで直し」や余分な変更が混ざっていないか（別コミットなら可）。原則 1 PR 1 変更。
- **マージ安全性**: マージ後に挙動が変わる点を洗い出し、その差が問題を起こさないと確証を持てるか。
- 本当に必要か／より単純な方法はないか／過剰設計（YAGNI）か／コストに見合うか。
- **テストも最小差分の一部**: 挙動を担保しない・過剰で CI を遅くするテストは指摘する（網羅不足は test-coverage が別途見る）。
- 設計が見落としているエッジケースは何か。

### communication（タイトル・説明・コミットの文章品質）
コードではなく**文章（タイトル＋説明・コミットメッセージ）**の品質を見る常時レンズ。観点本文は下の
「タイトル＋説明」節（**diff 突き合わせ手順を含む**）と「コミットメッセージ」節、および各層 REVIEW.md の
コミット具体基準に従う。ディスパッチ時はこれらをそのまま注入する。
- 出力は severity を付けず **communication 品質の改善点**として列挙する（説明: why / 可読性 / 簡潔さ / 整合、コミット: 型 / 件名 / 粒度 / why）。
- ただし**説明の主張が実装と矛盾し実害がある**ものだけは severity 付きで「コードの指摘」へ回す（出力フォーマット節の注記に従う）。

## タイトル＋説明（1つの文章として）

タイトルと説明は分割せず、連続した1つの文章として読む。

- **why/背景**: なぜ必要か（動機・課題）が読み取れるか。変更内容の言い換え（what）や手段の説明で止まっていないか。
  - 未使用リソース/コードの削除はメンテコスト削減という動機が自明なため、why が薄くても許容。
- **可読性**: 冗長で読みにくくないか。通して自然に読めるか（ログ全文貼付・定型文反復・一文過長等がないか）。
- **整合**: タイトルと本文に齟齬がないか。説明の主張が差分の実態と矛盾しないか。

### diff 突き合わせ（必須手順・「why あり」で止めない）

「why があるか」の確認だけで良しとしない。**説明の各文・各小節を diff と1つずつ突き合わせ**、その文が
**diff を見れば自明な what（ファイル追加・クラス名・フラグ名・既定値・「薄いアダプタ」等）だけを述べていないか**を判定する。

- diff から一目で分かる what しか書いていない文・小節は**冗長**。独立見出しを立てるほどの情報がなければ、削除 or 1行への降格を提案する。
- 残す価値があるのは、diff に現れない **why・設計意図・保証（no-op / 後方互換 / 失敗時挙動 / 影響範囲）・トレードオフ**を足している文。
- 見出し付き小節が差分規模に対して多すぎないか（薄い変更に小節を量産していないか）も可読性として見る。箇条書きへの集約を提案する。

> これは「why の有無」の**逆向きチェック**。why が伝わる簡潔な文には指摘しない（下記）一方、
> what の重複は why の有無と独立に冗長として拾う。両方を必ず回す。

簡潔で why が伝わる文章には指摘しない（過剰指摘の抑制）。

## コミットメッセージ（汎用枠組み）

- **why**: What の繰り返しで終わらず、なぜ必要かが伝わるか（typo・自明な削除等は省略可）。
- **件名**: 簡潔か。
- **粒度**: 1コミット＝独立してリバート可能なアトミックな単位か。

> 件名の文字数・型・言語・prefix などの**具体基準はプロジェクト固有**（`<project_root>/.claude/REVIEW.md` と参照先の規約に従う）。

## 帰属（attribution）— 変更起因に絞る

| 分類 | 定義 | 扱い |
|---|---|---|
| `IN_DIFF` | 変更行/追加行そのものへの指摘 | **採用** |
| `CHANGE_CAUSED` | 変更行外だが今回の変更が直接の原因（既存呼び出し元が壊れる / 意味波及で consumer の挙動・判定が変わる） | **採用**（検証で経路確認必須） |
| `PRE_EXISTING` | 変更と無関係な既存コードの品質指摘 | **ドロップ**。件数のみ記録 |

- 「変更ファイルの中だから関係ある」は因果ではない。変更行を起点に経路をたどれて初めて `CHANGE_CAUSED`。
- **スコープゲートは severity ゲートに優先**。`PRE_EXISTING` は申告 critical でも本文に戻さず件数記録のみ。

---

## 確信度フィルタ

各指摘を 0-100 で評価し、**80 未満は出力しない**。規約由来の指摘は、該当ルールが明文で禁止しているか確認してから出す。
2+ レンズが同じ箇所を指摘していれば確信度シグナルとして残す。

---

## severity 物差し（検証後の最終判定）

検証後の実害で一意に決める:

- データ破損 / 認可漏れ・テナント越境 / セキュリティ侵害 → **Critical**
- ユーザー可視の意図しない挙動変化（表示・出力・外部配信が変わる）→ 仕様意図が未確認なら **Critical**、軽微で確認余地があれば **Major**
- 性能劣化（出力データ自体は正しい）→ **Major**
- 可読性・命名・スタイル（挙動に影響しない）→ **Minor**

主観形容詞（「明白なバグ」等）はそのまま採用せず、事実だけ抽出して再判定する。
「仕様意図が未確認」の確認源は PR タイトル＋説明・spec/plan・外部仕様。確認できなければ安全側（Critical）に倒す。

## 出力フォーマット（日本語・簡潔）

下記の**記入済み例の形に倣う**。原則:
- コードの指摘（severity 軸）→ PR説明・コミットの指摘（改善点軸）の順で**別セクション**（説明の指摘がコードの Critical に埋もれないように）。
- 絵文字は使わず severity はテキスト見出し。**散文の長文解説は避け**、成果物（実行番号・最小 diff・改善後の文）は添える。


### ルール（例で表せない分）
- **コード**: severity は Critical → Major → Minor 順。`検証`（Read したソース）は critical / major のみ。`file:line` は **head 版の実際の行番号**（範囲は `:start-end`・「付近」等の曖昧表現は不可・不明なら Read で特定）。指摘ゼロなら「指摘なし」と 1 行。**指摘がすべて Minor なら各1行に畳む**（多文の解説にしない）。
- **確認・後続メモ**（severity なし・各1行）: 現在の実害が無い**安全確認**（見たが問題なし）や**将来PR向けの任意改善**は、severity 付きの「コードの指摘」に混ぜず**この枠に1行**で（「スキルは見たが今は不要」の信号）。0件なら節ごと省略。
- **PR説明・コミット**: 必ず `説明` / `コミット` の2グループに分ける。各項目は `<対象>｜<種別>: 診断` 形式（文言修正は `現状:` / `改善:` を別行・逐語は要点のみ `…` 省略）。グループが0件なら「指摘なし」と 1 行（clean シグナル・空欄にしない）。OK 項目の praise 列挙は書かない。
- 説明の主張が実装と矛盾し**実害**があるものは、ここでなく「コードの指摘」に severity 付きで載せる。
- **スコープ外**（`PRE_EXISTING`）は件数のみ。**ここで結果は終わり**（自己評価コメント等は付けない）。
